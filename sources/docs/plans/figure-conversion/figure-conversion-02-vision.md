# Figure Conversion Plan 02: Vision-Assisted Figure Interpretation

## Purpose

Add vision-assisted interpretation for figures that cannot be classified or reconstructed with sufficient confidence using local PDF extraction alone.

Vision is a targeted assistive step. It should not replace deterministic extraction, image fallback, or RK3-owned renderers. Vision models should return structured figure interpretations that RK3 can validate, score, review, and render.

## Location

Active plan file: `docs/plans/figure-conversion-02-vision.md`  
Feature standards file: `docs/features/figure-conversion-standards.md`

## Scope

Phase 2 adds:

- per-feature Figure Vision agent configuration
- vision request packaging
- structured vision response schema
- provider/model metadata tracking
- integration with classification and candidate generation
- Figures QA UI display of provider/model output
- side-by-side provider comparison support where practical

User correction workflows are Phase 3.

## Configuration

Figure Vision should be configured as a distinct task in the system-wide per-feature AI agent configuration.

Do not reuse generic QA vision configuration silently. Figure interpretation may need different provider/model choices, prompts, temperature, output schema, or cost controls than other visual QA tasks.

Suggested config key shape:

```json
{
  "features": {
    "figure_conversion": {
      "vision": {
        "enabled": true,
        "provider": "configured-provider",
        "model": "configured-model",
        "prompt_version": "figure-vision-v1",
        "max_image_side": 1600,
        "store_raw_response": true,
        "store_structured_response": true
      }
    }
  }
}
```

Adapt this to existing RK3 configuration conventions.

## When to use vision

Vision should be used selectively.

Good triggers:

- local classifier returns `unknown` with medium/high figure-likelihood
- local extraction finds a figure but cannot associate labels to shapes
- local extraction detects a chart but values are ambiguous
- local extraction detects a diagram but cannot determine structure
- user or project policy requests AI-assisted figure interpretation
- eval run requests provider comparison

Avoid vision for:

- simple figures already classified confidently
- figures selected for image fallback with no reconstruction goal
- photos or decorative illustrations unless alt text is needed
- documents where policy disables external AI

## Vision request package

Do not send only the image crop when local artifacts are available. Send a compact package containing:

- rendered figure crop
- page number
- figure bounding box
- detected title/caption/source/note candidates
- extracted text blocks with approximate coordinates
- vector primitive summary
- current local classification and confidence
- specific questions RK3 needs answered

The prompt should ask the model to resolve ambiguities and produce structured JSON, not final HTML.

## Structured response schema

Vision output should follow a strict schema.

Example shape:

```json
{
  "figure_kind": "stacked_bar_chart",
  "confidence": 0.86,
  "summary": "A stacked horizontal bar chart showing degrees of change across program areas.",
  "geometry_encodes_data": true,
  "data_values_visible": true,
  "elements": [
    {
      "role": "title",
      "text": "Degrees of Change"
    },
    {
      "role": "data_segment",
      "label": "Some program areas",
      "value": 25,
      "unit": "%",
      "color_hint": "orange"
    }
  ],
  "data": [],
  "recommended_render_modes": ["chartjs", "svg", "accessible_image"],
  "warnings": [],
  "needs_human_review": false
}
```

The exact schema should be versioned. Store the prompt version and schema version with every response.

## Validation rules

RK3 must validate vision output against local extraction.

Examples:

- If the model returns text not found in local text extraction, flag it.
- If the model returns data values that conflict with visible extracted text, flag it.
- If the model claims a chart type that conflicts with obvious vector geometry, flag it.
- If the model omits text visible in the crop, reduce text coverage score.

Vision output can improve classification confidence, but it should not be treated as automatically authoritative.

## Candidate generation after vision

After a vision response is validated, RK3 may:

- update figure classification
- update data-criticality flags
- create or refine extracted data
- create new render candidates
- add warnings
- require manual review
- change selected output if confidence threshold is met

Final rendering should still be generated by RK3 renderers.

## Provider metadata

Every AI-assisted figure interpretation must record:

- provider
- model
- configuration key
- prompt version
- schema version
- timestamp
- request hash
- response hash
- cost/usage metrics if available
- whether raw response was stored
- whether structured response passed validation

The Figures QA tab must show provider and model for each AI-assisted conversion.

## Side-by-side provider comparison

Because figure interpretation quality may vary significantly across providers, the system should allow side-by-side comparisons during QA/eval.

A practical implementation can support multiple vision interpretation records per figure:

```json
{
  "figure_id": "p5_fig2",
  "vision_interpretations": [
    {
      "provider": "provider-a",
      "model": "model-a",
      "prompt_version": "figure-vision-v1",
      "structured_response": {},
      "validation": {},
      "created_at": "..."
    },
    {
      "provider": "provider-b",
      "model": "model-b",
      "prompt_version": "figure-vision-v1",
      "structured_response": {},
      "validation": {},
      "created_at": "..."
    }
  ]
}
```

The production path may use one configured provider, while QA/eval paths may run comparison providers.

## Figures QA UI updates

Update the Figures tab to show:

- whether vision was used
- provider/model used
- prompt/schema version
- structured interpretation
- validation warnings
- comparison outputs if multiple providers were run
- whether vision changed classification or selected render mode
- whether vision created or modified extracted data

The UI should make it clear which conclusions came from local extraction and which came from vision.

## Testing and evaluation

Create a fixture set for vision-specific cases:

- ambiguous diagram where local classifier fails
- chart with visible values but complex layout
- figure with labels that need grouping
- figure where vision should recommend image fallback
- figure where providers disagree

For each fixture, store expected acceptable interpretations, not just one exact output. Evaluate:

- correct figure type
- correct data-criticality flags
- correct text coverage
- correct label-to-shape association
- hallucination avoidance
- useful warnings

## Acceptance criteria

Phase 2 is complete when:

- Figure Vision has a distinct per-feature AI config.
- Vision requests include figure crops plus local extraction context.
- Vision responses use a versioned structured schema.
- Provider/model metadata is stored per figure interpretation.
- Vision output is validated against local extraction.
- Vision can update classification and candidate generation without bypassing RK3 renderers.
- The Figures QA tab shows provider/model details and vision interpretation data.
- The system can store multiple vision interpretations for side-by-side provider comparison.
- A small vision eval fixture set exists.

## Non-goals

- No user correction UI.
- No direct raw vision-to-final-HTML workflow.
- No requirement to send every figure to vision.
- No guarantee that vision output is trusted without validation.
