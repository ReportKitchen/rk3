# Figure Conversion Plan 00: Early Figures QA Review UI

## Purpose

Add a Figures tab to the existing RK3 QA UI early in the Figure Conversion work. This tab does not need correction controls yet. Its purpose is to make RK3’s figure detection and interpretation visible while the extraction and rendering pipeline is being built.

The review UI is a prerequisite for useful development. Agents should not treat figure conversion as a black-box output problem. The UI should expose what RK3 thinks it found, where it found it, what evidence it used, and what output options it generated.

## Location

Active plan file: `docs/plans/figure-conversion-00-review-ui.md`  
Feature standards file: `docs/features/figure-conversion-standards.md`

## Scope

Build a Figures tab in the existing QA UI.

The first version should be read-only. Do not build correction workflows in this plan. Corrections are Phase 3.

## Required data surfaced in the UI

For each processed PDF, the Figures tab should show a list of detected figure candidates. Each candidate should expose:

- figure ID
- source page number
- figure index on page
- bounding box
- rendered figure crop
- page preview with figure bounds overlay
- any detected split between figure area and surrounding text area
- nearby title/caption/source/note candidates
- text blocks inside the figure region
- vector primitive summary where available
- classification result
- data-criticality flags
- available render candidates
- selected default render mode
- confidence scores
- warnings
- artifact paths
- AI provider/model metadata when an AI-assisted step has been used
- applied correction ops when correction support exists later

## UI behaviors

### Document-level view

Add a Figures tab alongside existing QA views. The tab should provide:

- figure count
- figure type distribution
- count of image fallback outputs
- count of reconstructed outputs
- count requiring review
- count with AI assistance
- filter by page
- filter by classification
- filter by render mode
- filter by warning/review status

### Figure list

Each row or card should show:

- small crop thumbnail
- figure ID
- page
- classification
- selected render mode
- confidence summary
- warning badges

### Figure detail panel

Selecting a figure should show:

1. Original page preview with overlayed bounds.
2. Figure crop.
3. Extracted text and candidate title/caption/source/note.
4. Vector primitive summary.
5. Classification and data-criticality result.
6. Candidate outputs and scores.
7. Selected output preview.
8. Artifact metadata.

The detail panel does not need editing controls in this plan.

## Overlay requirements

The page preview should support simple overlays:

- detected figure bounding box
- crop bounds
- title/caption/source/note candidate regions
- text blocks assigned to the figure
- text blocks excluded from the figure

The overlay can be approximate. It is more important that agents can see how the detection behaves than that the overlay is polished.

## Artifact requirements

The QA UI should read from persisted artifacts generated by the pipeline. Avoid coupling the UI to temporary in-memory state.

Minimum artifact shape:

```json
{
  "document_id": "example-document",
  "figures": [
    {
      "figure_id": "p3_fig1",
      "page": 3,
      "bbox": [0, 0, 100, 100],
      "crop_path": "artifacts/figures/p3_fig1.png",
      "classification": {
        "type": "unknown",
        "confidence": 0.0
      },
      "render_candidates": [],
      "selected_render_mode": "image",
      "warnings": []
    }
  ]
}
```

Use the actual RK3 artifact conventions where they already exist.

## Implementation notes

- Use the existing RK3 engine and UI conventions.
- Add libraries only when they materially improve extraction, rendering, or inspection.
- Do not require AI or vision to build this tab.
- Do not block the tab on high-quality classification. It is acceptable for early candidates to show `unknown`.
- The first version can show vector primitive counts before it shows full primitive overlays.

## Acceptance criteria

The plan is complete when:

- The existing QA UI has a Figures tab.
- A processed PDF with candidate figures shows figure rows/cards.
- Each figure has a stable ID, page number, crop, and bounding box.
- The UI shows a page preview with bounds overlay.
- The UI shows extracted text candidates when available.
- The UI shows classification and render candidate placeholders even if the classifier is still basic.
- The UI can display warnings and confidence values.
- The UI reads from persisted artifacts, not only transient runtime state.

## Non-goals

- No user correction workflow.
- No final rendering system required.
- No vision provider integration required.
- No promise that all figures are detected correctly.
- No requirement to publish reconstructed figures from the UI.
